Value Story02 · The problem

Two chronic problems with AI-assisted dev

Problem 1

Amnesia

The AI starts every session from a blank slate — no durable project memory, context re-explained again and again.

Problem 2

Ungoverned

Its work is ad-hoc, untracked, and unauditable. No record of what changed or why.

Problem 3

Inconsistent

Output differs per tool and per developer, with no shared lifecycle or quality gates.

The result: speed without a paper trail — hard to trust, hard to audit, hard to scale to a team.
The stakeholder hook. AI pair-programmers are fast, but they forget everything between sessions and leave no consistent record. Two engineers on the same repo get different conventions, context, quality. For a team or client engagement, that untracked, ungoverned work is a real risk — not a productivity story. This is the pain we're about to remove.

Architecture & Memory06 · Invariant

The one load-bearing rule: files are the single source of truth

  • No database anywhere — even live agent runs are files + ephemeral server memory.
  • Both the AI tools and the Cockpit read and write the same .tcgstackflow/ files.
  • The folder doubles as an Obsidian vault — browsable, graph-viewable Markdown.
  • The rule was never "read-only" — it was "no second store" — which is exactly why the Cockpit could safely become read-write (ADR 0032).
Pivot into technical depth — engineers lean in. Every decision bends to one rule: plain files are the only store, no parallel database (ADR 0004/0020/0024). That's what makes the workspace portable, auditable, diff-able. Subtle point: "read-only" was never the rule; "no second store" was — which is why the Cockpit could start writing task files without violating anything.

Architecture & Memory08 · Memory

Memory that compounds — the LLM-wiki + qmd

  • Flat Obsidian-flavoured Markdown, heavy [[wikilinks]], a Map-of-Content index.md — Karpathy's LLM-wiki pattern.
  • Operations vocabulary: Ingest / Query / Lint, plus a workspace Audit.
  • qmd is the mandatory hybrid search layer: keyword + vector + LLM re-rank (~2 GB local models).
  • Only the Ingester writes to the wiki — one ingest may touch 10–15 pages.
  • A git pull-digest hook auto-feeds upstream changes to the Ingester on every git pull — capturing the meaning of work, not just a file list.
The heart of reason #1. The wiki is the project's brain, kept token-efficient and current by the Ingester after every finished task — so memory literally gets smarter each cycle. qmd (Tobi Lütke's tool) is the mandatory discovery layer every agent uses, installed and indexed at init. The append-only log.md uses a locked grep-able prefix. The pull-digest hook is the quiet superpower — a git pull becomes an Ingester feed capturing the meaning of upstream work.
Architecture & Memory09 · Tasks

The two-file task system

The plan

TASK details {ID}.md

Subtasks + sizes + acceptance criteria. Human-facing.

The record

TASK {ID}.md

Append-only YAML log: timestamp · author · summary · files · why · validation · tags.

  • Every task is exactly two files — non-negotiable, never per-subtask files.
  • The load-bearing detail: the author: field records which tool did each piece (claude / codex / copilot) — a task can be planned by Claude and coded by Codex.
  • Tasks move active/ → completed/ → archive/; the strict rule keeps history machine-readable.
Two files, always. Plan is human-facing; log is an append-only YAML audit trail. The load-bearing detail is the author field — one task can be planned by Claude and coded by Codex, and the log records exactly who did what. That's how portability and auditability coexist. Managers hear: every task carries its own record of what changed and why, automatically.

Agents & Lifecycle11 · Two gates

Two gates most workflows collapse into one

Static gate

Is the code right?

Reads the diff. Correctness by reading.

→ Reviewer · IN_REVIEW
→
Dynamic gate

Does it work?

Runs the suites + the app. Correctness by running.

→ Tester · IN_TEST → VALIDATED

Nothing reaches the wiki as "done" until it has both passed review and been proven to run. For refactor tasks, the Reviewer's scope-drift blocker relaxes and the Tester becomes the real gate — prove behavior is unchanged.

A design point engineers appreciate. Static review (correctness by reading) and dynamic verification (correctness by running) are genuinely different jobs — separate roles, separate task states. Most AI workflows blur them; here "right" and "works" are two gates a task must pass in order. For stakeholders: nothing reaches the wiki as done until it passed review AND was proven to run.

Cockpit & Governance14 · Run mechanics

Launch, watch, and run to completion

  • Press ▶ Run — spawns headless claude, streams token-by-token over SSE. Closing the tab does not kill the run.
  • Continuation loop: iter 0 sends the role prompt; later iters --resume the session until the task advances or a 6-iteration cap.
  • Chain mode (⛓ run-to-completion): auto-advances coder → reviewer → tester → ingester until INGESTED.
  • Concurrency: 1 active run per project (the slot is the lock), unbounded across projects.
  • Crash-resilient: orphaned runs reconciled to "aborted at pause point" on reboot.
continuation loop
iter 0 → role prompt
↻ resume
iter n → --resume session
↻ … up to 6
✓ task advances → next role
Show the mechanics engineers will scrutinize. The continuation loop is the normal mode, not a retry loop — iter 0 sends the role prompt, later iters --resume until the agent advances the task or hits the 6-iteration cap, tokens accumulating into one run record. Chain mode walks an unattended task through the pipeline with a backward-bounce limit that defuses the classic coder↔reviewer infinite loop. The in-memory active slot IS the per-project lock — no lockfile — and a killed/crashed run never silently advances a task.
Cockpit & Governance15 · Governance

Live governance — four risk levels, one modal

LOWjust do it
MEDIUMdo it, log it
HIGHask first
CRITICALask first + rollback plan
Approve action? CRITICAL
Actionrm -rf ./legacy && migrate
Whydrop deprecated module
Rollbackgit revert <sha>
  • HIGH/CRITICAL pause the running agent and long-poll (no timeout) for a browser Approve/Deny — each decision recorded to the task log.
  • Fails closed: unknown/mcp__* tools default HIGH; a compound command takes the max segment's level; rules can only raise. CRITICAL can't be one-clicked (HTTP 428 without a rollback ack).
  • Degrades explicitly per tool — shown as a badge: mcp-intercept / hook-command / sandbox-preset / none.
The safety headline — dwell here for stakeholders. Governance went from an informally-followed doc (ADR 0008) to machine-enforced at runtime (ADR 0027): a risky action physically blocks the agent mid-flight until a human decides in the browser, and "Deny" is non-fatal — recorded as deferred to human, the run continues with what it can. Engineers care that it's fail-closed by construction: can't reach the Cockpit → deny; unknown tools → HIGH; trusted prefix + && rm -rf still maxes at CRITICAL. Reduced governance on a lesser tool is always a visible badge, never silent.

Cockpit & Governance17 · Reality

Grounded in reality — Jira drift & signal-driven tasks

  • Two statuses per task: workspace status (our lifecycle) vs Jira status (client business truth) — the Cockpit flags drift.
  • Credential-free: sync-jira (Atlassian MCP) writes a local jira-cache.json; the server only reads it — it never holds a credential.
  • The Planner never fabricates — can't fetch a Jira ticket → it stops and asks.
  • External signals become tracked tasks: /tcgflow-task-from-snyk · -cypress · -datadog — dedup + group at the source's natural unit.
  • Built-in rituals: weekly Tempo timesheet, Jira sync, wiki lint, workspace audit.
Shows the tool lives in a real delivery environment, not a vacuum. Workspace status and Jira status can legitimately diverge, and the Cockpit surfaces that drift rather than hiding it — the server never holds a credential, it just reads a cache the AI-mediated sync skill wrote. The signal-to-task family (Snyk, Cypress, Datadog) turns production noise into tracked two-file tasks with severity mapped to risk. Reinforce no-fabrication: the Planner would rather stop and ask than invent a ticket's contents.

Maturity & Close20 · Roadmap

Where it's going — cost-spreading & new tools

  • Runner-adapter seam (ADR 0035) — a pure interface. Adding Codex/Copilot is a parse+argv adapter, not a rewrite.
  • Ship order by fidelity: Claude (full parity) → Copilot (best governance, degraded stream) → Codex (sandbox-governed) → Antigravity (Copy-prompt).
  • Cost-spreading thesis: a per-role tool map lets Claude plan/review while a cheaper model codes — shipped defaulted OFF until savings are measured.
  • Next rung: team-usable → OSS-ready (MIT, npm geekstackflow) — earning each complexity before adding it.
The roadmap is architecture, not wishlist. The runner-adapter seam already exists and is unit-tested with a fake adapter; today only Claude is registered and an unknown tool is safely refused with a 501. The economic thesis — Claude plans/reviews, a cheaper executor codes against a tight plan — is baked into the per-role tool map but defaults to all-Claude until real savings are measured. That "earn the complexity" rhythm is the through-line; the next earned rung is full team use, then OSS.
