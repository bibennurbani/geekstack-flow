# A per-tool runner-adapter seam; fidelity degrades explicitly, never silently (builds the adapter ADR 0025 deferred)

ADR 0025 chose a per-role tool map for cost-spreading ("Claude plans, Codex/Antigravity codes against a tight plan") and stated, line 47, that "each tool's headless flags/sandbox/auth differences are **encapsulated behind a small per-tool 'runner adapter'** in the Orchestrator; adding a tool is a new runner adapter, not a core change." It deferred two follow-ups (line 48): concurrency (settled by ADR 0026) and **in-run governance per tool**. This ADR fixes the runner-adapter seam and that per-tool governance question together.

The promise was never built. In code, `readRoleTool` (run.cjs) reads `claude|codex` from `orchestrator.roles`, but its single caller rejects anything but `claude` with a `501 runner-not-implemented`, and the executor's transport is **fused to Claude's CLI**: `spawnOnce` hard-codes `-p --output-format stream-json --include-partial-messages`, the resume mechanic assumes `--resume <session_id>`, the governance gate assumes `--mcp-config`/`--permission-prompt-tool`, and `chat()` repeats the same Claude flag string. So the cost-spreading goal that *motivated* the Orchestrator is unreachable — the config knob is real, the behaviour behind it is a `501`. `readRoleTool` is a pass-through reader (delete it and only the 501 vanishes); the behaviour that should sit behind an interface (per-tool argv, stream parse, resume idiom, gate model) sits behind none.

## What the four runner tools can actually do (verified June 2026)

Before committing the interface we verified each tool's *real* headless surface against official docs and the open-source CLI repos (no-fabricate-on-unreachable-source rule; marketing ignored, sources cited in the research record). The result is that "works just like Claude" is achievable **per axis, per tool** — not uniformly — which is exactly why the adapter must *declare* capabilities rather than assume Claude's.

| Axis the Continuation loop needs | Claude Code | Codex (`codex exec`) | GitHub Copilot (`copilot -p`) | Antigravity (`agy -p`) |
|---|---|---|---|---|
| Headless subprocess | ✅ `claude -p` | ✅ `codex exec` | ✅ `copilot -p` | ⚠️ exists, but capturing stdout yields 0 bytes (open bug #76) |
| Machine-readable stream | ✅ incremental NDJSON | ✅ incremental NDJSON (`--json`) | ⚠️ JSONL; incremental delivery unconfirmed | ❌ plain text only |
| Per-turn token usage | ✅ parseable `usage` | ✅ parseable | ⚠️ session-total only (undocumented RPC) | ❓ unknown |
| Session resume | ✅ `--resume` | ✅ `codex exec resume` | ✅ `--resume` | ❌ no headless session-id (open bug #7) |
| External permission hook (the Cockpit card) | ✅ `--permission-prompt-tool` (MCP intercept) | ❌ not in `exec`; only via the *experimental* app-server JSON-RPC (open hang bug #11816) | ✅ `preToolUse` hook (**fail-closed**) — never `permissionRequest` (**fail-open**) | ❌ none |
| **Verdict** | **full parity** | **parity except the approval card** | **governance parity; stream/tokens degraded** | **not viable headlessly — Copy-prompt only** |

## Decision

- **The seam is a pure `RunnerAdapter` module** — no I/O, no `child_process`; the executor still owns spawning, so the continuation loop stays unit-testable with a fake adapter. Interface: `buildSpawn(run, ctx) -> { bin, args, env }` (ctx carries `prompt`, `iter`, `resumeId`, `governance`, `mode: 'run' | 'chat'`), `parseLine(line, state) -> Event[]` mapping the tool's stream to uniform `{ delta | tokens | session | result }` events, and `resumeIdFrom(state) -> string | null`. The loop, **Run-token** accumulation, **Status safety-net**, **Auto-advance chain**, inactivity timeout and `chat()` all consume the seam and are tool-agnostic; `readRoleTool` becomes the **selector**.
- **Adapters declare `capabilities`**, and the Orchestrator honours **fidelity per tool, degrading explicitly**: `gate: 'mcp-intercept' | 'hook-command' | 'sandbox-preset' | 'none'`, `tokens: 'per-turn' | 'session-total' | 'none'`, `stream: 'incremental' | 'jsonl' | 'text' | 'none'`, `resume: boolean`, `topology: 'we-spawn' | 'we-host-jsonrpc'`. The Cockpit renders the run's `gate` as a **badge** (e.g. *"approval card active"* vs *"sandbox-governed"*); reduced governance is never silent. `tokens: 'session-total'` is acceptable because Run tokens are recorded **once per Run** (ADR 0033), not per turn.
- **No drivable headless mode → Copy-prompt, not a faked Run.** A tool the Orchestrator cannot drive as a subprocess remains a Copy-prompt handoff target (already in doctrine) and graduates to a live runner only when it ships the surface.
- **Ship order, by fidelity not preference:** (1) **Claude** — the reference adapter, shipped first as a behaviour-preserving refactor of today's executor (full parity, repays the ADR 0025 debt, makes the loop testable immediately). (2) **GitHub Copilot** — best governance parity (the fail-closed `preToolUse` hook drives the Cockpit card), shipped *degraded* behind an empirical stream/incremental smoke test; tokens session-total. (3) **Codex** — full/near parity on stream/tokens/resume via `codex exec`; `coder: codex` runs are governed by Codex's own `--sandbox workspace-write --ask-for-approval` **preset policy** with a *"sandbox-governed"* badge, **not** the approval card. (4) **Antigravity** — Copy-prompt only; defer the runner adapter until `agy -p` emits a captured stream and a headless session-id.

## Considered options

- **Per-tool `gate`, parity-first, degrade explicitly with a visible badge** — *chosen*. Keeps the cost-spreading goal reachable now (Codex on coder tasks) while never weakening governance silently; the badge makes fidelity legible to the user.
- **Approval card or it is not a runner** (card mandatory for every Run) — rejected: it would defer Codex alongside Antigravity and kill the cost-spreading goal for no safety gain over the sandbox preset + visible badge. Reconsider only if a sandbox-preset run is later judged unacceptable for any HIGH/CRITICAL coder work.
- **Build Codex's approval-card path now** via the experimental app-server JSON-RPC host topology (`execCommandApproval`/`applyPatchApproval`) — deferred: it is explicitly experimental, inverts who spawns whom (a second runner code path), and has an open hang-forever bug (#11816, no approval timeout). Revisit when it stabilises; the `topology: 'we-host-jsonrpc'` flag reserves the seam for it.
- **Keep Claude-only, no adapter** — rejected: leaves ADR 0025:47 unbuilt and the founding cost-spreading goal a `501`.

## Consequences

- The continuation loop becomes testable with a fake adapter (drive synthetic events, assert advance/continue/abort) — independent of any real tool, closing the gap that today only two of the executor's paths are exercised through a real subprocess.
- **cwd pinning (Claude):** `--resume` lookup is scoped to the run's directory and its git worktrees, so every resume iteration must spawn with iteration-0's cwd. The executor already spawns with `cwd: run.project_path`; the adapter must not re-derive cwd from the (possibly higher) multi-repo workspace root.
- **Governance fidelity is now an explicit, recorded property of a Run**, not an assumption. A Codex coder run carries `gate: sandbox-preset` and the badge; the per-Run governance record (ADR 0027/0008) notes the gate model so an auditor sees how the run was governed. The fail-open `permissionRequest` path (Copilot) is documented as forbidden — only the fail-closed `preToolUse` hook may drive the card.
- **Budget pricing follows the role's tool.** This composes with consolidating the duplicated spend/budget computation: once an adapter is selected, the pricing model is the role's actual tool rather than the hard-coded `claude-opus` literal.
- **No new store; files-as-truth intact (ADR 0024).** The adapter is pure; run state stays files + ephemeral server memory. Sequential-within-project (ADR 0026) is unchanged — the seam is about *which* tool a Run drives, not concurrency.
- **Capability flags, not this table, are the runtime source of truth.** These tools change fast (Codex's `--json` schema is still firming; Antigravity's CLI shipped days before this review). The adapter declares what it can do today; when a tool gains an axis (e.g. Antigravity exposes a headless session-id, or Codex's app-server stabilises), it is a capability bump in one adapter, not a core change — which is the whole point of the seam.
